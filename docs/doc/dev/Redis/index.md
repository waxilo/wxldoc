## Redis为什么这么快

### 内存存储：物理速度的起点

**所有数据都在内存中**，避免了磁盘 I/O（即使是 SSD，延迟也在微秒级；而内存是纳秒级）。

#### 单线程 + I/O 多路复用

##### 1. **单线程模型（核心命令执行）**

- Redis 6.0 之前：**完全单线程**处理网络 I/O + 命令执行。
- Redis 6.0+：**网络 I/O 多线程**（读写 socket），但**命令执行仍是单线程**。
##### 为什么单线程反而更快？

- **无锁设计**：避免多线程的锁竞争、死锁、上下文切换开销。
- **CPU 不是瓶颈**：Redis 操作主要是内存读写，速度极快，单核 CPU 足以支撑 10w+ QPS。
- **代码简单、逻辑清晰**：减少 bug，提升稳定性。
##### 2. **I/O 多路复用（epoll/kqueue）**

- 允许**一个线程同时监听成千上万个客户端连接**。
- 只有当某个 socket “就绪”（可读/可写）时，才通知主线程处理。
- 类比：餐厅只有一个服务员，但每桌有“呼叫铃”——只服务“举手”的客人，绝不空等。
### 高效的数据结构：为性能量身定制

| 数据类型       | 底层结构                         | 性能优势                          |
| ---------- | ---------------------------- | ----------------------------- |
| 字符串        | SDS（Simple Dynamic String）   | O(1) 获取长度、防溢出、二进制安全、内存预分配     |
| 哈希/列表/集合   | ziplist / quicklist / intset | 小数据紧凑存储，节省内存                  |
| 有序集合（ZSet） | Skip List（跳表）                | O(log N) 插入/查询，支持范围操作，比红黑树更简单 |
| 全局键空间      | dict（哈希表）                    | O(1) 查找，支持渐进式 rehash（避免卡顿）    |

### 异步持久化：不阻塞主线程
虽然数据在内存，但 Redis 支持持久化（RDB 快照 / AOF 日志）：
- **fork() 子进程**执行磁盘写入（利用 Copy-on-Write 机制）
- **主线程继续处理请求**，几乎无阻塞
- AOF 重写、RDB 生成均由后台完成
